# UI languages (i18n)

WasmHatch's interface is translated with [Lingui](https://lingui.dev/). The
model's replies are a separate concern: the agent answers in whatever language
the user writes in (with the UI language as the fallback), regardless of the
interface language — that rule lives in the system prompt, not in catalogs.

## How it fits together

- English source strings stay in the code, wrapped in Lingui macros
  (`<Trans>…</Trans>` in JSX, `` t`…` `` elsewhere).
- [`src/lib/locales.ts`](../src/lib/locales.ts) is the locale registry — the
  single source of truth for which languages exist. The picker, catalog
  loading, `<html lang>`, and the translation script all derive from it.
- [`src/lib/i18n.ts`](../src/lib/i18n.ts) resolves the active locale:
  explicit choice in Settings (persisted in `localStorage`) beats
  `navigator.languages` matching (script-aware: `zh-TW` → `zh-Hant`,
  `pt-PT` → `pt-BR`) beats English. Switching languages re-renders in place —
  no reload, so running hatchlings are never interrupted.
- Catalogs are one `src/locales/<locale>.po` per language. The Vite plugin
  compiles them on the fly; only the active locale's chunk is loaded.
- Chrome's built-in model only speaks en/ja/es/de/fr
  ([`src/lib/builtin-ai-language.ts`](../src/lib/builtin-ai-language.ts));
  other UI languages keep the built-in provider working with English output,
  and cloud providers answer in the user's language.

## Editing or adding strings

1. Write the English string in code with a macro.
2. `npm run i18n:extract` — updates every `.po` with the new entries.
3. Fill the new entries in every `.po` (by hand or by asking an agent — see
   below), then `npm run i18n:check`. Translation fixes are welcome as plain
   PRs — the files are ordinary gettext catalogs.

Guidelines: never build sentences by concatenating translated fragments
(make one message per full sentence variant), use `plural()` for counts, and
put variables in named placeholders.

## Adding a language

1. Add one entry to `UI_LOCALES` in `src/lib/locales.ts` (BCP 47 code +
   native name).
2. `npm run i18n:extract` to create the catalog.
3. Fill the catalog and verify with `npm run i18n:check`.
4. If the language needs a script-specific system font stack, add a
   `:lang()` rule in `src/styles.css` next to the existing CJK/Thai ones.

RTL languages (ar, he, fa, ur) additionally need the CSS logical-properties
pass and `dir` wiring — that work is scoped but not done yet; see the plan in
the repo history before adding them.

## How translations are produced (`npm run i18n:check`)

Translations are written by a maintainer or by an AI agent working in the
repo (an earlier Anthropic-API script was dropped — no API key is involved),
and reviewed as a normal git diff. `scripts/check-catalogs.mjs` is the
machine-checkable part: it verifies every catalog is complete, `{placeholder}`
sets survive verbatim, and ICU plural messages keep their skeleton, variable,
`other` branch, and `#` count. Nothing translation-related runs at app
runtime.

Conventions for whoever (or whatever) translates:

- Never translate brand and protocol names: WasmHatch, MCP, API, URL, ZIP,
  Chrome, Ollama, `OLLAMA_ORIGINS`, `VITE_GOOGLE_CLIENT_ID`, `drive.file`.
- "hatchling" is the product mascot (a just-hatched chick that works for
  you): use one consistent, warm rendering per language (ja: ハッチリング)
  and keep the hatching metaphor in "Hatch a new one" / "Hatching workspace".
- "ticket" is a work item on a shared board (kanban sense); "workspace" is
  the user's file area in the browser; "artifact" is a generated HTML
  deliverable — prefer the natural word for "deliverable" over a
  transliteration where that reads better.
- ICU plural messages use the plural categories the target language actually
  needs (e.g. ru/uk/pl add `few`/`many`; zh/ja/ko/th/vi/id keep only
  `other`).
- Voice: concise, warm, plain UI language for a general audience; imperative
  for buttons; Google product names follow each language's official Google
  naming.

`ja` is maintained by hand first — treat it as the reference for tone.

## Tests

- Unit: locale negotiation (`src/lib/locales.test.ts`) and the built-in-model
  language intersection (`src/lib/builtin-ai-language.test.ts`).
- E2E runs with the browser locale defaulting to English, so text-based
  selectors keep working; a dedicated smoke test switches to `ja` and checks
  `<html lang>` plus a translated string.
